# Implementation Plan: module_src_gnu.c_25

## Summary

Port `src/gnu.c` into a focused Rust module that preserves the existing output behavior of `gnu_output_handler` without expanding scope beyond the current file and function set.

The Rust implementation should:
- migrate the logic in `gnu_output_handler` into a single Rust module aligned with the current project layout;
- replace C-style manual memory handling with ownership and borrowing from the Rust standard library;
- convert implicit C error paths into explicit `Result`-based handling where the surrounding Rust code requires it;
- keep control flow and data formatting close to the original implementation to reduce migration risk.

The overall approach is a direct source migration: map the existing function into Rust with minimal structural change, introduce only the Rust data types needed to represent the current anonymous C data usage, and validate behavior with targeted `cargo test` coverage.

## Data Model

The analysis identifies an `anonymous` data structure. Since C anonymous structs/unions often appear as file-local or inline state carriers, the Rust mapping should be driven by actual field usage in `gnu_output_handler`.

### Data Structure Mapping
- `anonymous` -> a private Rust `struct` or `enum` in `src/gnu.rs`, chosen by observed usage:
  - use `struct` if the C anonymous type is a fixed field container;
  - use `enum` only if the anonymous form encodes tagged alternatives in practice.

### Mapping Rules
- C pointers to owned buffers -> `String` or `Vec<u8>` depending on textual vs byte-oriented use.
- C borrowed string pointers -> `&str` where UTF-8 is guaranteed by surrounding Rust code, otherwise `&[u8]` or validated conversion at the boundary.
- Nullable pointers -> `Option<T>` / `Option<&T>`.
- Integer flags -> `bool` or small integer types (`u32`, `i32`) only where boolean conversion is not semantically correct.
- Output destinations represented by `FILE *`-style usage -> generic writer interfaces such as `&mut impl std::io::Write` if the surrounding port already uses writer abstraction; otherwise use the project’s existing output abstraction.

### Memory Management Decisions
- Eliminate manual allocation/free pairs; ownership should be explicit in local variables and returned values.
- Avoid long-lived mutable aliasing; prefer immutable borrows with short mutable write sections.
- If the C code uses temporary formatting buffers, replace them with stack-local formatting via `write!`/`format!` as appropriate.

### Error Handling Decisions
- Replace C sentinel return/error patterns with `Result<(), std::io::Error>` or a project-local error type if one already exists in the Rust port.
- Preserve observable failure points, especially output write failures.
- Avoid introducing panic-based control flow for ordinary runtime conditions.
