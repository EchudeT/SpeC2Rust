## Overview

This module is the main operational unit for the `shc` program. It parses command-line options, reads an input shell script, derives or applies a keying context, transforms script-related data through an RC4-style stream operation, generates C source output containing embedded protected data, and invokes a build step to produce the final artifact.

The Rust rewrite must preserve the observable behavior of this module as a command-line driven script-to-generated-source/build pipeline. The specification is limited to functionality evidenced by `src/shc.c` and does not introduce new capabilities.

## Scope

The Rust version of `module_src` must implement these functional areas evidenced in `src/shc.c`:

- command-line argument parsing and dispatch
- script file reading
- key initialization and key derivation from file metadata
- byte-stream transformation of buffers using the module’s ARC4-style routine
- shell text syntax validation through shell evaluation
- random/noise byte generation used during emitted-source preparation
- formatted byte-array/source emission
- generation of C output from processed inputs
- orchestration of generation and build steps
- process entrypoint behavior equivalent to the current module flow

## Out of Scope

## Feature Specification

### 1. Command-line driven module execution

The module accepts process arguments, interprets them, stores option state, and runs the overall generation workflow. Argument handling is split between single-argument parsing and full-argument traversal (`parse_an_arg`, `parse_args`, `do_all`, `main`).

The Rust version must:

- accept CLI arguments as the primary input mechanism for this module
- parse supported options and positional values in a way that preserves current module behavior
- reject or fail on invalid argument combinations in the same execution stage as the original module
- drive the full module workflow from parsed inputs through program entry

### 2. Script ingestion and validation support

The module reads the source shell script from a file path (`read_script`) and contains a shell-evaluation helper (`eval_shell`) used to evaluate shell text.

The Rust version must:

- read the complete script content from the specified file input
- surface failure when the script cannot be read
- preserve shell-evaluation behavior used by the module to check or process shell text when that path is exercised
- treat script text as data to be later embedded/generated by the pipeline

### 3. Key state setup and data transformation

The module maintains an internal ARC4-style state with an initialization routine (`stte_0`), key scheduling (`key`), stream transformation (`arc4`), and optional file-derived keying (`key_with_file`).

The Rust version must:

- initialize transform state before keying or stream use
- accept arbitrary byte sequences as key material
- transform byte buffers in-place or equivalently so that the same input state and keying sequence produce the same resulting bytes as the C module
- support deriving key material from file metadata as currently used by the module
- report failure when file-based key derivation cannot be completed

### 4. Randomized filler/noise support for emitted content

The module includes bounded random generation (`rand_mod`, `rand_chr`) and writes randomized noise into buffers (`noise`).

The Rust version must:

- generate bounded pseudo-random values sufficient to preserve current module behavior
- generate single random characters as used by the module
- populate buffers with randomized noise according to the original call patterns and parameters
- preserve whether noise generation is intended for string-like or raw-byte output where evidenced by the `noise` interface

### 5. C-source array emission

The module formats raw bytes into textual C representations (`prnt_bytes`, `prnt_array`, `dump_array`) and writes the generated C source file (`write_C`).

The Rust version must:

- emit byte data in C-source-compatible array form
- preserve the distinction between printing chunks of bytes and printing named arrays
- support output formatting sufficient for the generated C file to serve the same role as in the original module
- write the generated source artifact to the requested output path
- fail when the target output cannot be written

### 6. Build orchestration

The module performs a build step after generation (`make`) and coordinates the full process (`do_all`).

The Rust version must:

- execute the same high-level sequence of operations as the original module
- invoke the build phase after successful generation
- propagate build failure as module failure
- stop subsequent stages when prerequisite stages fail

## Requirements

### Functional Requirements

#### FR-1: Parse command-line inputs
The module shall parse process arguments and derive the option/state values needed for execution, using behavior traceable to `parse_an_arg`, `parse_args`, `do_all`, and `main` in `src/shc.c`.

#### FR-2: Enforce argument-driven control flow
The module shall determine whether and how the generation workflow runs based on parsed arguments, with invalid inputs preventing normal workflow continuation. Traceable to `parse_an_arg`, `parse_args`, `do_all`, and `main`.

#### FR-3: Read script content from a file
The module shall load script text from a filesystem path into memory for later processing. Traceable to `read_script`.

#### FR-4: Evaluate shell text when requested by the workflow
The module shall support execution/evaluation of shell text in the same functional role as the original helper. Traceable to `eval_shell`.

#### FR-5: Initialize stream-cipher state
The module shall provide a state reset/initialization step required before key scheduling or stream transformation. Traceable to `stte_0`.

#### FR-6: Apply key material to internal state
The module shall accept caller-provided key bytes and mix them into internal transformation state. Traceable to `key`.

#### FR-7: Transform byte buffers using the module’s stream operation
The module shall transform supplied byte buffers according to the initialized/keyed state in a way functionally equivalent to the original implementation. Traceable to `arc4`.

#### FR-8: Support metadata-derived keying from a file
The module shall derive and apply keying input from file metadata and report success or failure. Traceable to `key_with_file`.

#### FR-9: Produce bounded random values for generation support
The module shall generate bounded random integers and single random characters for internal generation support. Traceable to `rand_mod` and `rand_chr`.

#### FR-10: Populate buffers with randomized noise
The module shall write randomized filler bytes/characters into a caller-supplied region according to supplied bounds/flags. Traceable to `noise`.

#### FR-11: Format bytes as C-compatible textual output
The module shall print byte content in textual form suitable for generated C source emission. Traceable to `prnt_bytes`, `prnt_array`, and `dump_array`.

#### FR-12: Generate C source output from processed module data
The module shall write a C source artifact containing the generated/embedded data needed by the overall tool. Traceable to `write_C`.

#### FR-13: Run the build step after successful generation
The module shall invoke the build phase once prerequisite generation succeeds. Traceable to `make` and `do_all`.

#### FR-14: Propagate failures across stages
The module shall expose failure from argument parsing, file reading, key derivation, generation, shell evaluation, or build so that the overall run does not incorrectly report success. Traceable across `parse_an_arg`, `read_script`, `key_with_file`, `eval_shell`, `write_C`, `make`, `do_all`, and `main`.
